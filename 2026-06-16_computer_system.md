# 운영체제
## 정의
운영체제(OS: Operating System) 는
**컴퓨터 시스템의 여러 자원을 효율적으로 운영 및 관리함으로써 사용자에게 컴퓨팅 시스템을 편리하게 사용할 수 있는 환경을 제공해 주는 시스템 소프트웨어**
로 사용자와 컴퓨팅 시스템 간의 중개 역할 수행.
## 주요 구성 요소
자원을 효율적으로 관리해 한정된 자원으로 빠르게 작업을 처리할수 있도록 돕는 **커널(kernel)**,
사용자가 편리하게 작업할 수 있게 환경을 제공하는 **사용자 인터페이스(UI: User Inteface)**
로 구성.
### 커널
컴퓨팅 시스템의 자원을 관리하고, 사용자 프로그램을 제어함
### 사용자 인터페이스
커널에 사용자의 명령을 전달하고 실행 결과를 사용자에게 알려 줌
- CLI (Command Line Interface)
  --
  - 사용자가 직접 문자열로 된 명령어를 입력하는 방식
  - 응답 속도는 빠르나 명령어 습득이 어려움
- GUI (Graphic User Interface)
  --
  - 사용자가 명령어를 모르더라도 마우스를 통해 아이콘과 같은 그래픽 이미지를 선택해 작업 수행
  - 현재 대부분 컴퓨터에서 사용
- NUI (Natural User Interface)
  --
  - 사람의 손이나 몸 동작 등 행동이나 음성을 이용해 명령을 입력하는 방식
  - 얼굴/음성/모션 인식 등 인공지능 기술이 적용되어 새로운 인터페이스 기술로 발전중
### 컴퓨터용 운영체제
 - Window
   --
   - Microsoft 사에서 개발한 OS
   - GUI 를 활용해 사용자에게 편리한 환경을 제공하며 호환성이 높음
   - 프로그램이 다소 무겁고, 업그레이드 때마다 빠른 CPU 와 큰 메모리 요구
- macOS
  --
  - Apple 사에서 개발한 매킨토시 전용 OS
  - 같은 회사의 여러 기기가 간편하게 통신 가능
  - 공용 클라우드에 데이터를 저장할 수 있어 편리함
  - 애플사가 아닌 다른 회사 제품과 호환성이 낮음
- Linux
  --
  - Linus Torvalds 가 개발한 오픈 소스 운영체제, 누구나 자유 사용
  - Unix 기반으로 안정적이며 다양한 배포판이 있어 사용자 편의에 따라 무료 사용
  - 소스 코드가 공개되어 있어 많은 사람이 참가해 기술을 꾸준히 개발해 안정적임
  - PC 및 대형 컴퓨터에서 주로 사용
  - CLI 외에 GUI 도 지원
### 모바일 운영체제
 - Android
   --
   - Google 사에서 개발한 오픈 소스 기반의 모바일 운영체제
   - 무료로 제공해 많은 모바일 기기 업체들이 사용
 - iOS
   --
   - 아이폰에 탑재된 모바일 운영체제
   - 애플사의 제한적인 정책으로 애플사 제품에만 사용
### 임베디드 운영체제
 - 내비게이션, 스마트워치, 스마트 냉장고 같이 소형 공간에 내장해 동작하도록 만든 OS
 - 작은 크기의 CPU나 메모리, 배터리 등의 제한된 환경에서 최적화되게 만들어짐
## 처리 방식
### 일괄 처리 시스템
 - Batch processing
 - 처리할 데이터를 일정량 또는 일정 기간 동안 모았다가 한꺼번에 처리
 - 대량의 작업 처리에 적합
   - 작업이 순차적으로 실행돼 응답 시간이 길어질 수 있음
   - 모든 작업을 한꺼번에 처리해 프로그램 실행 중간에 사용자가 데이터를 입력하거나 수정 불가능
 - 주로 채점/추첨/요금 청구서 업무 등에 사용
### 다중 프로그래밍 시스템
 - Multiprogramming
 - 여러 개의 프로그램을 주기억 장치에 적재하고 한 개의 CPU를 이용해 실행
 - CPU의 유휴 시간이 줄어 단일 처리 시스템에 비해 효율성이 높음
   - 메모리에 여러 개의 프로그램을 적재해 메모리를 관리해야 함
   - 먼저 실행할지 결정하는 **스케줄링** 필요
### 시분할 시스템
 - Time Sharing
 - 다중 사용자 환경에서 사용
 - 여러 사용자에게 CPU의 시간을 분할해서 할당하고 번갈아 처리함
 - 혼자서 CPU를 독점하고 있는 것처럼 사용 가능
 - 전체적인 작업의 처리량과 응답 시간 최소화 가능
### 실시간 처리 시스템
 - Real-Time System
 - 정해진 시간 내에 빠르게 작업을 수행하고 완료하는 데 초점을 둠
 - 시간에 제약이 있는 작업을 처리하고, 완료 시간이 중요하고 예측할 수 있어야 함
 - 주로 은행/예약/항공기 제어 업무 등에 사용

# 프로세스
## 개념
프로그램은 보조 기억 장치에 저장되어 있다가 호출되면 주기억 장치에 할당되어 작업을 처리하는 데, 이때 할당된 프로그램 **(실행중인 프로그램)**
> 프로그램은 프로세스 단위로 주기억 장치에 적재되고 프로세서인 CPU를 할당받아 실행될 수 있음
## 상태 변화도
 - 생성 `프로세스가 생성된 상태`
   - 준비 `프로세스가 메모리에서 준비 중인 상태`
   - *디스패치 (무엇을 실행할 지 선택)*
     - 실행 `프로세스가 CPU에서 실행되고 있는 상태`
     - *입출력 요청 (block)*
       - 대기 `프로세스가 특정 자원이나 이벤트를 기다리는 상태)`
       - *입출력 완료 (wake up)*
         - **준비**
     - *타임아웃*
       - **준비**
     -
     -
       - 완료 `프로세스가 실행을 완료한 상태`
|기존||이후|설명|
|:-|:-|:-|:-|
|생성|→|준비|생성 상태에서 메모리가 할당됨|
|준비|→|실행|준비 상태에서 CPU를 할당됨|
|실행|→|준비|실행 상태에서 CPU 할당 시간이 만료됨|
|실행|→|대기|실행 상태의 프로세스가 입출력 등의 요청이 발생됨|
|대기|→|준비|대기 상태에서 요청한 일이 완료됨|
|실행|→|완료|실행이 완료되면 프로세스는 할당받은 모든 자원을 반납함|
## CPU 스케줄링 기법
### FCFS
 - 선입 선처리 (First Come First Served)
 - 가장 단순한 비선점 알고리즘
 - CPU를 요청하는 순서대로 할당하며 큐로 구현
 - 일괄 처리 시스템에선 매우 효율적
   - 빠른 응답을 요청하는 대화식 시스템에선 부적합함
> - 새로운 프로세스가 들어옴
>   - 큐의 마지막에 연결
>     - 차례가 되면 큐의 앞부분에 있던 프로세스가 CPU를 할당 받고 큐에서 삭제
### SJF
 - 최단 작업 우선 (Shortest Job First)
 - CPU 이용 시간의 길이가 가장 짧은 프로세스부터 시행
 - 평균 대기 시간을 줄이는 데 유리함
   - 긴 작업이 계속 뒤로 밀릴 수 있어 기아 현상이 생길 수 있음
### RR
 - 라운드 로빈 (Round Robin)
 - 시분할 시스템에서 사용하는 가장 대표적인 선점 스케줄링 방식
 - 프로세스가 한 번에 실행 가능한 시간인 **시간 할당량(time slice)** 정의
 - 시간 할당량이 끝나면 인터럽트를 발생시켜 다른 프로세스에게 CPU를 할당해 모든 프로세스가 공정하게 실행되게 함
 - 구현을 위해 **선입 선출(FIFO)** 로 작동하는 큐를 순환 큐로 설계
 - 시간 할당량을 초과하는 작업 시간을 가진 프로세스는 실행을 마친 후 다시 큐의 마지막 위치로 이동
   - 프로세스가 CPU를 일정 시간 동안 사용하고 다른 프로세스엑 넘겨 줘야 해서 호위 효과가 줄어듬
   - 시간 할당량이 너무 길거나 너무 짧으면 시스템 성능이 저하됨
 - **타임아웃(timeout)** - 프로세스가 정해진 시간 할당량을 초과하는 것
 - #### 시간 할당량이 너무 긴 경우
 - FCFS 방식처럼 동작해서 현재 프로세스는 작업을 완료할 충분한 시간을 얻게 되지만, 다른 프로세스는 그만큼 기다려야 하므로 평균 대기 시간이 늘어난다.
 - #### 시간 할당량이 너무 짧은 경우
 - 여러 개의 프로세스가 빠르게 병렬 실행되지만 문맥 교환이 자주 일어나 시스템 전체 성능이 저하된다.
### PS
 - 우선순위 스케줄링 (Priority Scheduling)
 - 프로세스가 큐에 도착하면, 도착한 프로세스의 우선순위와 현재 실행 중인 프로세스의 우선순위를 비교해 가장 높은 프로세스에게 CPU를 할당
 - 우선순위가 동일한 프로세스들은 선입 선출 순서로 스케줄링
 - 우선순위 스케줄링은 선점형과 비선점형
   - **선점형** 새로 도착한 프로세스의 우선순위가 현재 실행하는 프로세스의 우선순위보다 높으면 CPU를 선점
   - **비선점형** 새로 도착한 프로세스의 우선순위가 현재 실행 중인 프로세스보다 높아도 CPU를 선점하지 못함
## 교착 상태
 - **교착 상태(deadlock)** 둘 이상의 프로세스가 다른 프로세스가 점유하고 있는 자원을 무한정 기다리며 진행이 중단된 상태
### 교착 상태 발생 조건
 - 상호 배제
   --
   - 한 프로세스가 사용 중인 자원을 동시에 사용할 수 없음
 - 점유와 대기
   --
   - 프로세스가 하나 이상의 자원을 점유한 상태에서 다른 자원을 기다리는 상태
 - 비선점
   --
   - 다른 프로세스가 점유하고 있는 자원을 강제로 빼앗아 선점할 수 없는 상태
 - 환형 대기
   --
   - 자원을 할당받은 두 프로세스가 서로 할당된 자원을 요청하고 있는 상태
   - 자원 할당 그래프의 원의 형태
### 교착 상태 해결 방법
 - 교착 상태 예방(prevention)
   --
   - 발생 조건 중 하나 이상을 제거해 발생을 사전에 막는 방법
   - 예방 방법은 자원 낭비가 심하고 기아 상태를 초래할 수 있음
   - 상호 배제 조건 제거
     --
     - 시스템 내의 모든 자원을 공유해 사용
     - 시스템에는 공유할 수 없는 자원이 존재해 현실적인 문제 발생
   - 점유와 대기 조건 제거
     --
     - 프로세스가 자원을 점유한 상태에서 다른 자원을 기다리지 못하게 함
     - 프로세스가 필요한 자원을 한꺼번에 요청해 실행에 필요한 모든 자원을 점유하거나 아무것도 점유하지 못하게 함
     - 당장 필요하지 않은 자원을 점유해 자원의 낭비 발생
   - 비선점 조건 제거
     --
     - 모든 자원을 선점할 수 있도록 만듬
     - 공유하는 자원의 접근 순서에 따라 실행 결과가 달라지는 경우엔 마음대로 자원의 접근 순서를 바꾸는 건 문제 발생
       - 프린터 등
   - 환형 대기 조건 제거
     --
     - 점유와 대기 중인 프로세스들이 환형을 이루지 못하도록 만듬
     - 모든 자원에 번호를 부여하고 자원을 한 방향으로만 사용하도록 설정
 - 교착 상태 회피(avoidance)
   --
   - 시스템의 운영 상태를 살펴보며 교착 상태 발생 가능성을 피해가는 방법
   - 안전 상태, 불안전 상태로 나눌 수 있음
   - **안전 상태** 자원 할당이 적절히 이루어져 교착 상태가 없음
   - **불안전 상태** 교착 상태 발생 가능성이 있음
     - 은행원 알고리즘으로 불리우는 교착 상태 회피 알고리즘을 이용해 시스템의 상태를 안정 상태로 유지
       - 자원의 사용률을 떨어뜨릴 수 있음
       - 교착 상태 발생 가능성을 시뮬레이션해야 함
         - 시스템의 전체 자원 수, 각 프로세스가 앞으로 사용할 자원 수 등의 정보를 미리 파악해야 함
         - **현실적으로 한계**
 - 교착 상태 탐지(detection) 및 회복(recovery)
   --
   - 이미 발생한 교착 상태를 탐지 알고리즘이나 자원 할당 그래프의 사이클 유무 등을 이용해 찾아내 복구함
   - 선점을 통한 회복이나 프로세스 강제 종료를 통한 회복을 함
   - 교착 상태에 있는 모든 프로세스를중단하면 교착 상태는 회복하지만 연산을 다시 해야 해서 시간과 비용이 많이 듬
     - 시간과 비용 감소를 위해 희생자 선택 원칙에 따라 한 개씩 중단하면서 회복되었는지 확인해야 함

# 메모리
## 반입 전략
보조 기억 장치에 저장된 프로그램이나 데이터를 언제 주기억 장치로 가져올 것인지 결정함
### 요구 반입
 - 실행 중인 프로그램의 참조 요청에 따라 보조 기억 장치의 프로그램이나 데이터를 주기억 장치로 적재
 - 요청할 때마다 적재해야 하므로 대기 시간 발생
### 예상 반입
 - 실행 중인 프로그래에 의해 참조될 프로그램이나 데이터를 예측해 미리 주기억 장치로 적재함
 - 사용 가능성이 높은 대상을 예측하는 것이 어려움
 - 예측하여 가져온 부분이 사용되지 않을 수 있음
## 배치 전략
반입된 프로세스와 데이터를 주기억 장치의 어떤 부분에 배치할지 결정함
### 최초 적합
 - first-fit
 - 프로세스를 첫 번째 가용 공간에 배치 (검색 최소화, 빠른 할당)
 - 검색은 가용 공간 리스트의 맨 앞이나 이전에 검색이 끝났던 곳에서 시작
 - 가용 공간을 빨리 찾으나 공간 활용률이 떨어질 수 있음
### 최적 적합
 - best-fit
 - 프로세스를 가용 공간 중에서 프로세스 할당 후 남는 공간이 가장 작은 곳에 배치
 - 가용 공간이 크기순으로 정렬되어 있지 않으면 전체 검색 필요
 - 가용 공간 이용률은 향상되나 계속 정렬하는 과정이 필요해 할당 과정에 많은 시간 소요
### 최악 적합
 - worst-fit
 - 프로세스를 가용 공간 중에서 프로세스 할당 후 남는 공간이 가장 큰 곳에 배치
 - 최적 적합과 마찬가지로 공간이 크기순으로 정렬되어 있어야 함
## 교체 전략
주기억 장치의 기억 공간이 충분하지 않을 때, 현재 적재된 프로세스 중 제거할 프로세스를 찾아 교체
### 선입 선출
 - FIFO (First In First Out)
 - 가장 오래된 페이지를 먼저 교체함
 - 구현이 간단하며 페이지 관리를 위해 준비 큐 사용
 - 주기억 장치에 도착한 순서대로 새로운 페이지는 큐의 맨 뒤에 저장
 - 주기억 장치에 새로운 페이지를 적재하기 위해 준비 큐에서 가장 먼저 도착한 페이지를 교체
### 최근 최소 사용
 - LRU (Least Recently Used)
 - 현 시점에서 가장 오랫동안 사용하지 않은 페이지를 교체
   - 시간 지역성에 의해 최근에 가장 많이 사용한 페이지가 앞으로도 가장 많이 사용될 것이라고 판단
 - 우수한 교체 알고리즘으로 사용되지만, 최근 참조 여부로 교체 순서를 결정해 스택이나 계수기를 이용
### 최소 사용 빈도 수
 - LFU (Least Frequently Used)
 - 참조 횟수가 가장 작은 페이지를 교체
 - 각 페이지마다 참조 횟수 계수기가 필요
 - 초기 단계에서 많이 참조된 페이지가 이후에는 사용되지 않을 때 불리
## 단편화
프로세스와 데이터가 주기억 장치의 기억 공간을 차지하고 사용할 수 없을 만큼 작은 공간이 남는 현상
(연속 메모리 할당에서 발생하며, 단편화가 많을수록 주기억 장치의 사용 가능한 공간이 줄어 기억 공간이 낭비됨)
### 내부 단편화
 - 프로세스가 들어가고 공간이 남을 때
 - 프로세스가 실제로 사용하는 공간보다 더 큰 메모리 공간을 할당받아 남는 공간에서 발생
### 외부 단편화
 - 프로세스가 공간보다 더 클 때
 - 프로세스가 실제로 사용하는 공간보다 할당받을 메모리 공간이 더 작아서 할당받지 못할 때 발생
### 해결 방법
단편화된 공간들을 통합해서 가용 공간으로 만들어야 함
 - 메모리 통합
   --
   - 하나의 작업이 끝났을 때 다른 빈 공간과 인접해 있는지 검사하여 하나로 통합
   - 메모리 전반에 흩어진 공간을 모두 통합하기는 어려움
 - 메모리 압축
   --
   - [디스크 조각 모음?]()
   - 사용 중인 공간과 가용 공간을 적절히 배치해 여러 개의 가용 공간을 하나의 큰 가용 공간으로 만듬
   - **쓰레기 수집(GC - gabage collection)** 라고도 불림
   - 메모리를 압축하는 동안 시스템은 다른 일을 중지해야 하며, 시스템의 자원 소모가 큼
     - 자주 실행할 수 없음
## 가상 메모리
보조 기억 장치의 일부 공간을 주기억 장치처럼 사용하는 것으로 실행할 프로그램이나 데이터를 보조 기억 장치의 가상 공간에 연속적으로 저장하고 실행헤 필요한 페이지만 주기억 장치의 프레임에 적재하여 실행
 - 이를 활용해 주기억 자치의 크기와 관계없이 여러 개의 프로그램이나 데이터를 실행할 수 있음
 - **가상 공간** 주기억 장치처럼 사용되는 보조 기억 장치의 일부 공간
 - **페이지** 가상 공간을 고정된 크기로 분할한 영역
 - **프레임** 실 공간을 페이지 크기와 같게 분할한 영역
 - **세그먼트** `페이지와 다르게 다른 크기로도 분할이 가능한 것`
### 작동 절차
 - 가상 공간은 페이지라는 블록으로 분할, 주기억 장치의 실 공간 또한 같은 크기의 프레임을 분할
 - 페이지 내 실행할 프로세스에 부여된 가상 주소를 사상 테이블에 대응하는 주기억 장치 프레임의 실 주소로 변환
   - 사상 (mapping) 과정 수행
 - 주기억 장치의 프레임에 실행할 페이지가 적재되어 있으면 CPU를 통해 실행
 - 주기억 장치의 프레임에 실행할 페이지가 없거나 더 이상 페이지를 적재할 프레임이없으면 페이지 교체 알고리즘으로 페이지를 교체한 후 실행
### 장점
 - 프로세스의 크기와 상관없이 실행 가능
 - 각 프로세스에 연속된 가상 주소 공간을 제공
 - 프로그램 하나가 여러 공간에 분산해 적재돼 주기억 장치를 좀더 효율적으로 활용
### 단점
 - 주기억 장치와 보조 기억 장치 사이에 이동량이 증가해 스와핑 공간이 필요
 - 페이지 교체 알고리즘과 같이 요구된 프로세스의 페이지가 없을 때 처리하는 적절한 방법 필요
### 스와핑
 - 주기억 장치에서 실행되고 있는 프로세스가 실행 중에 임시로 보조 기억 장치와 같은 예비 저장 장치로 내보내졌다가 실행을 계속하기 위해 다시 주기억 장치로 돌아오는 과정
 - 스왑 아웃 (swap out)
   --
   - 보조 기억 장치로 이동하는 것
 - 스왑 인 (swap in)
   --
   - 주기억 장치로 돌아오는 것

# 디스크 스케줄링
다중 프로그래밍 기업에서 여러 프로세스가 필요한 프로그램이나 데이터를 요청하면 큐를 이용해 요청을 저장하고, 데이터에 접근하기 위한 경로를 결정하는 것
> 탐색 시간을 최적화하기 위해 수행되며 처리량을 극대화하고 응답 시간을 줄여 평균 응답 시간의 차이가 크지 않도록 함
 - FCFS
   --
   - 선입 선처리 (First Come First Served)
   - 요청한 순서대로 처리되는 가장 간단한 기법
   - 구현하기 쉽고, 어떤 요청도 무기한 연기되지 않아 공정성을 유지
 - SSTF
   --
   - 최소 탐색 시간 우선 (Short Seek TIme First)
   - 현재 헤드 위치에서 탐색 거리가 가장 가까운 요청을 먼저 처리함
   - 디스크 접근 시간을 실질적으로 감소시키지만 상대적으로탐색 거리가 먼 요청이 무기한 연기되는 기아 상태 발생
 - SCAN
   --
   - 스캔 (SCAN)
   - 현재 헤드 위치에서 진행 방향을 가지며 진행 방향 내에서 가장 가까운 거리에 잇는 요청을 먼저 처리
   - 입출력 헤드가 디스크 한쪽 끝에 도달 시, 다시 역방향으로 이동해 처리
   - SSTF의 단점인 응답 시간 편차를 줄일 수 있음
 - C-SCAN
   --
   - 순환 스캔 (Circular-SCAN)
   - 헤드가 한 방향으로 이동하면서 요청을 처리하지만 한쪽 끝에 도착하면 반대편의 다른 끝으로 이동하여 남은 요청을 처리하며, 이동 방향이 바뀌지 않음
   - 트랙 안쪽과 바깥쪽을 차별하지 않는 방법
