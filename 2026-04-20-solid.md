# SOLID 원칙
## SRP — Single Responsibility Principle
하나의 클래스는 **하나의 책임**만 가져야 함

- 책임이 많을수록 변경 시 영향 범위가 커짐
- 사용자 정보 관리 + 로그인 -> X / 각각 분리 -> O

---

## OCP — Open-Closed Principle
**확장에는 열려** 있고, **수정에는 닫혀** 있어야 함

- 기존 코드를 건드리지 않고 기능을 추가할 수 있어야 함
- 다형성과 인터페이스 활용이 핵심

## LSP — Liskov Substitution Principle
**자식 클래스는 부모 클래스를 대체**할 수 있어야 함

- 부모 타입 자리에 자식 객체를 넣어도 동작이 같아야 함
- 부모: 증가 기능 -> 자식: 감소로 구현 -> X

## ISP — Interface Segregation Principle
인터페이스는 **클라이언트에 맞게** 잘게 분리

- 하나의 거대한 인터페이스보다 작은 인터페이스 여러 개가 좋음
- 불필요한 기능에 의존하지 않도록 설계

## DIP — Dependency Inversion Principle
구체 클래스가 아닌 **추상화(인터페이스)에 의존**

- 구현체에 직접 의존 -> 변경에 취약
- 인터페이스에 의존 + DI로 주입 -> 유연성 증가

## Clean-Up
- **SRP** | 하나의 책임만 가지기
- **OCP** 수정 없이 확장 가능하게 만들기
- **LSP** 자식은 부모를 대체할 수 있어야 함
- **ISP** 인터페이스는 작게 나누기
- **DIP** 추상화에 의존
