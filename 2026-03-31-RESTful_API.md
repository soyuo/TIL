# 1. RESTful API란?

*RESTful API*는 REST 라는 아키텍처 스타일을 따르는 API이다

*URL + HTTP* 메서드로 자원을 조작하는 방식이다

### 기본 개념

* **자원(Resource)** → URL로 표현


* **행위(Action)** → HTTP 메서드로 표현

  * GET → 조회
  * POST → 생성
  * PUT / PATCH → 수정
  * DELETE → 삭제

### 예시

```
GET    /users        → 사용자 목록 조회
GET    /users/1      → 1번 사용자 조회
POST   /users        → 사용자 생성
DELETE /users/1      → 삭제
```

**URL은 목적, 행동은 HTTP 메서드**

---

## 그래서 왜 쓰이는 가?

### 1) 웹 표준(HTTP)을 그대로 활용

* 이미 존재하는 HTTP 규칙을 그대로 쓴다
* 별도 규칙을 만들 필요가 없다

### 2) 프론트/백엔드 분리

* React, Vue 같은 프론트
* Node.js, Spring 같은 백엔드
  → 독립적으로 개발이 가능하다

### 3) 다양한 플랫폼 지원

* 웹, 모바일, IoT 등 어디서든 접근 가능하다

### 4) 직관적 구조

* URL만 봐도 어떤 API인지 짐작할 수 있다

---

# 3. REST의 핵심 특징

### 1) Stateless (무상태)

* 서버는 클라이언트 상태를 저장하지 않는다
* 요청마다 필요한 정보 다 포함한다

### 2) Client-Server 구조

* 역할 분리 (UI vs 데이터 처리)

### 3) Uniform Interface (일관된 인터페이스)

* URL 규칙 + HTTP 메서드 규칙을 유지한다

### 4) Cacheable

* 응답 캐싱 가능하다. (성능 향상)

---

# 4. 장점

### 1) 직관적이고 이해하기 쉬움

* URL + 메서드만 보면 기능 파악이 된다

### 2) 확장성 좋음

* 서버/클라이언트 독립 → 유지보수가 쉽다

### 3) HTTP 기반이라 범용성 높음

* 브라우저, 앱, 서버 어디서든 사용할 수 있다

### 4) 캐싱 가능 → 성능 향상

* GET 요청 캐싱이 가능하다

### 5) 학습 비용 낮음

* HTTP만 알면 기본 사용이 가능하다

---

# 5. 단점

### 1) 과도한 규칙 = 설계 스트레스

* “이게 RESTful한가?” 고민이 많이 생긴다

### 2) Over-fetch / Under-fetch 문제

필요한 데이터보다 많이 받거나 부족하게 받는다

이 문제 때문에 GraphQL 같은 대안이 등장했다

### 3) 상태 없는 구조의 한계

로그인 상태 관리 등은 따로 처리가 필요하다
* (JWT, 세션 등)

---

# 요약

RESTful API는
**“HTTP 기반으로 자원을 URL로 표현하고, 메서드로 조작하는 표준적인 API 설계 방식”**
